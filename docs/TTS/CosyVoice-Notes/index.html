Learning Map

先建立地图，再进入论文细节

CosyVoice 系列可以按一个主问题理解：如何让 TTS 同时具备内容一致性、音色克隆能力、低延迟、多语种泛化和真实场景鲁棒性。

你需要抓住的主线

  • CosyVoice 1：先把 speech token 做“语义化”，解决 token 和文本对齐不足的问题。
  • CosyVoice 2：把离线系统改造成可流式系统，同时简化 tokenizer 和 LM 结构。
  • CosyVoice 3：把实验室能力推向真实世界：多语种、长尾文本、跨语种、情绪和后训练。
  • Flow Matching：不是理解文本的模块，而是把语义条件转成连续 Mel 轨迹的声学生成器。

System Pipeline

四个模块各司其职

如果只记一句：LLM 负责“生成离散语音语义序列”，Flow Matching 负责“把离散语义变成连续声学表示”，Vocoder 负责“把声学表示还原成波形”。

01 / Tokenizer

把声音压成 token

CosyVoice 1 使用 ASR encoder + VQ 得到监督语义 token；CosyVoice 2 转向 FSQ；CosyVoice 3 引入 MinMo 多任务 tokenizer。

02 / LLM

预测语义 token

输入文本、prompt token 和条件信息，输出 speech token。它解决“说什么”和“像谁说”的离散序列问题。

03 / Flow Matching

从 token 走到 Mel

学习一个速度场，把噪声逐步推向目标 Mel。CosyVoice 2 的关键是 chunk-aware CFM，用于流式合成。

04 / Vocoder

从 Mel 还原波形

最后把连续声学特征变成可播放音频。它不负责理解文本，但决定声音细节和听感上限。

Flow Matching

用动态图理解“噪声如何走向语音”

Flow Matching 的核心不是“去噪很多步”这个表象，而是学习一个条件速度场：在文本、speaker prompt、speech token 的约束下，每一步应该往哪里移动。

Evolution

CosyVoice 1 → 2 → 3：每一代解决一个更现实的问题

版本演进不是简单堆模型，而是从“可做”到“可实时交互”，再到“真实复杂输入可用”。

CosyVoice 1

监督语义 token：让文本和语音 token 对齐

  • 重点解决无监督 speech token 语义不足、内容一致性弱的问题。
  • 形成 LLM + Conditional Flow Matching + Vocoder 的两阶段骨架。
  • 查看论文笔记 →
CosyVoice 2

流式化：把离线合成改造成实时系统

  • 引入 FSQ、预训练 LLM、统一流式 LM 和 chunk-aware CFM。
  • 核心工程目标是降低首包延迟，同时尽量保持质量无损。
  • 查看论文笔记 →
CosyVoice 3

真实场景：多语种、长尾文本与后训练

  • 引入 MinMo、DiffRO、百万小时级数据和更系统的 CV3-Eval。
  • 关注 in-the-wild 的输入复杂度，而不只是标准 benchmark。
  • 查看论文笔记 →

序列生成

声学生成

Conditional Flow Matching / DiT

实时系统

后训练

DiffRO / Gumbel-Softmax

理解如何把不可微的奖励目标变成可优化路径，以及 reward balance 的风险。

真实输入

Text Normalization / Pronunciation Inpainting

理解真实文本不是干净 benchmark，数字、缩写、多音字和纠音都会影响 TTS。
